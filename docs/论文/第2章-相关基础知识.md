# 第2章 相关基础知识

## 2.1 推荐系统概述

### 2.1.1 推荐系统基本概念

推荐系统（Recommender System）是一种用于解决信息过载问题的信息过滤技术，其核心任务是根据用户的历史行为、个人偏好以及物品的属性特征，从海量候选物品中筛选出用户可能感兴趣的物品子集，并以个性化列表的形式呈现给用户[1]。与搜索引擎不同，推荐系统不需要用户主动输入明确的查询关键词，而是通过分析用户的隐式行为（如浏览、点击、选择）和显式反馈（如评分、点赞）自动推断用户的兴趣偏好。

一个典型的推荐系统通常由三个核心模块构成：用户建模模块负责收集和表示用户偏好；推荐算法模块根据用户模型与物品模型计算预测评分或排序；结果输出模块将推荐结果以列表形式展示并收集用户反馈。推荐问题的数学形式可以描述为：设用户集合为 $U$，物品集合为 $I$，推荐系统的目标是学习一个预测函数 $f: U \times I \rightarrow \mathbb{R}$，使得 $f(u,i)$ 能够表示用户 $u$ 对物品 $i$ 的预测偏好程度，系统据此对候选物品进行排序，选择得分最高的 $K$ 个物品推荐给用户。

### 2.1.2 推荐系统主要方法分类

根据推荐策略的不同，主流的推荐系统方法可分为以下三类[1]：

（1）协同过滤推荐（Collaborative Filtering，CF）。协同过滤基于"物以类聚，人以群分"的基本假设，利用用户群体的行为数据进行推荐。其核心思想是找到与目标用户行为相似的其他用户，根据相似用户的偏好来预测目标用户的偏好。协同过滤不依赖于物品的内容特征，仅利用用户-物品交互矩阵（如评分矩阵或行为矩阵），因此具有良好的领域适用性。

（2）基于内容的推荐（Content-Based Recommendation）。基于内容的推荐从物品自身的属性特征出发，通过分析用户过去喜欢的物品的内容特征，构建用户兴趣画像，然后将待推荐物品的特征与用户画像进行匹配。这类方法不依赖其他用户的数据，因此不存在物品冷启动问题，但要求物品具有可解析的内容特征。

（3）混合推荐（Hybrid Recommendation）。混合推荐将多种推荐方法以某种策略进行组合，以弥补单一方法的不足，发挥各种方法的优势。常见的混合策略包括加权融合、切换式混合、特征组合和分层混合等[2]。本文提出的自适应混合推荐算法即属于加权融合策略的范畴，通过根据用户行为信号强度动态调整各子模型的权重，实现协同过滤、内容匹配与行为反馈的优势互补。

## 2.2 协同过滤推荐

### 2.2.1 基于用户的协同过滤

基于用户的协同过滤（User-Based Collaborative Filtering）是协同过滤中最经典的方法之一[7]。其基本思想分为两个步骤：首先计算目标用户与其他所有用户之间的相似度，找到与目标用户兴趣最相近的 $K$ 个邻居用户；然后根据邻居用户对目标物品的评分或行为，加权预测目标用户对该物品的偏好程度。

设用户 $u$ 与用户 $v$ 的行为向量分别为 $\vec{u}$ 和 $\vec{v}$，用户相似度为 $sim(u,v)$，则用户 $u$ 对物品 $i$ 的预测得分 $p(u,i)$ 可通过以下公式计算：

$$
p(u,i) = \frac{\sum_{v \in N(u)} sim(u,v) \cdot r(v,i)}{\sum_{v \in N(u)} |sim(u,v)|}
$$

其中 $N(u)$ 表示用户 $u$ 的邻居用户集合（只考虑那些对物品 $i$ 有过交互行为的邻居），$r(v,i)$ 表示用户 $v$ 对物品 $i$ 的行为得分。基于用户的协同过滤的优点在于能够发现用户潜在的兴趣领域，推荐结果具有一定的惊喜度（Serendipity）；其缺点在于当用户数量庞大时，相似度计算的复杂度较高，且新用户面临冷启动问题。

### 2.2.2 余弦相似度

余弦相似度（Cosine Similarity）是协同过滤中计算用户间相似度最常用的方法之一[8]。它将每个用户的行为向量视为高维空间中的一个向量，通过计算两个向量之间夹角的余弦值来衡量用户的相似程度。对于两个用户 $u$ 和 $v$，其余弦相似度的计算公式为：

$$
cos(u,v) = \frac{\vec{u} \cdot \vec{v}}{\|\vec{u}\| \times \|\vec{v}\|} = \frac{\sum_{i \in I_{uv}} r_{u,i} \times r_{v,i}}{\sqrt{\sum_{i \in I_u} r_{u,i}^2} \times \sqrt{\sum_{i \in I_v} r_{v,i}^2}}
$$

其中 $I_u$ 和 $I_v$ 分别表示用户 $u$ 和用户 $v$ 有过行为记录的物品集合，$I_{uv}$ 表示两个用户共同交互过的物品集合，$r_{u,i}$ 表示用户 $u$ 对物品 $i$ 的行为得分。余弦相似度的取值范围为 $[0, 1]$（在非负向量空间中），值越大表示两个用户的偏好越相似。当两个用户没有任何共同交互物品时，余弦相似度为 $0$。

余弦相似度的一个优势在于它对向量的模长不敏感，仅关注向量的方向（即偏好的相对分布），因此能够在一定程度上缓解不同用户评分尺度差异带来的影响。然而，当两个用户的共同交互物品数量很少时，余弦相似度可能会高估用户的相似性——例如，两个用户仅在一种物品上有共同交互且得分恰好相同，余弦相似度将等于 $1$，但这很可能是一种偶然现象。

### 2.2.3 收缩因子修正

针对小重叠度下余弦相似度可能产生假性高相似的问题，本文系统引入了收缩因子（Shrinkage）对用户相似度进行修正[12]。收缩因子的基本思想是：当两个用户的共同交互物品数量较少时，对相似度施加惩罚，使其向零收缩；随着共同交互物品数量的增加，修正后的相似度逐步逼近原始值。修正公式为：

$$
adjusted\_sim(u,v) = sim(u,v) \times \frac{overlap}{overlap + \lambda}
$$

其中 $overlap$ 表示两个用户的共同交互物品数量，$\lambda$ 为收缩系数（本文系统中取 $\lambda = 3.0$）。当 $overlap = 1$ 时，修正系数为 $1 / (1+3) = 0.25$，原始相似度被大幅削弱；当 $overlap = 10$ 时，修正系数为 $10 / (10+3) \approx 0.77$，相似度被适度保留；当 $overlap$ 趋于无穷大时，修正系数趋于 $1$，修正后的相似度等价于原始相似度。该机制使得只有在拥有足够共同交互证据支撑的情况下，两个用户才会被认定为高度相似，从而提升了协同过滤推荐的可靠性。

## 2.3 基于内容的推荐

基于内容的推荐方法从物品自身的属性特征出发，通过构建用户兴趣画像并与物品特征进行匹配来生成推荐[13]。与协同过滤不同，基于内容的推荐不依赖于其他用户的行为数据，而是利用物品的内容属性（如文本描述、类别标签、属性特征等）和用户的历史偏好记录来进行推荐。

基于内容的推荐通常包含三个步骤。第一步是物品特征提取，即将物品的属性信息转化为结构化的特征向量。在本系统中，每一道菜品具有两种关键的内容特征：餐食类型（mealType），取值为儿童餐、标准餐、清真餐、素食四类之一；口味标签集合（flavorTags），包括清淡、咸香、微辣、甜口、低脂、高蛋白共六种口味。第二步是用户画像构建，即根据用户的历史交互记录或显式偏好设置，生成用户的兴趣特征向量。在本系统中，用户的兴趣画像来源于两个方面：用户在偏好采集页面主动设定的餐食类型偏好和口味偏好，以及用户历史行为日志中累积的菜品交互得分。第三步是匹配度计算，即将候选物品的特征向量与用户兴趣画像进行匹配，计算用户对物品的兴趣度得分。在本系统中，匹配度的计算考虑了餐食类型的匹配奖励（单偏好 +0.45 / 多偏好 +0.38）和口味标签的重叠度奖励（$\min(0.35, 0.15 + matchedCount \times 0.10)$）。

基于内容的推荐的主要优势在于：不依赖其他用户数据，因此不存在物品冷启动问题；推荐结果具有较好的可解释性，可以明确告知用户推荐理由（如"符合您的清淡口味偏好"）；能够有效推荐新上架的物品或长尾物品。其主要局限在于：受限于物品特征的丰富程度，难以挖掘用户的潜在兴趣；推荐结果可能缺乏多样性，容易形成"信息茧房"效应。

## 2.4 混合推荐方法

混合推荐方法通过将多种推荐策略以一定方式进行组合，取长补短，以克服单一推荐方法的局限性[2]。常见的混合策略包括以下四种：

（1）加权融合（Weighted Fusion）。为每种推荐方法赋予一个权重系数，最终推荐得分由各方法得分的加权和决定：$S_{final} = w_1 S_1 + w_2 S_2 + \cdots + w_n S_n$。加权融合的关键在于权重的确定方式——可以是固定权重（如基于经验设定），也可以是自适应权重（如根据用户行为特征动态调整）。

（2）切换式混合（Switching Hybrid）。根据当前场景的条件（如用户是否有足够的历史行为数据），在多种推荐策略之间进行切换。例如，当用户行为信号充足时采用协同过滤，而当用户为新用户（冷启动场景）时切换为基于内容的推荐。

（3）特征组合（Feature Combination）。将多种推荐策略产生的中间特征拼接或组合，作为统一模型的输入特征。例如，将协同过滤产生的用户相似度特征与基于内容推荐的物品匹配度特征拼接后输入到元学习器中进行二次学习。

（4）分层混合（Meta-Level Hybrid）。使用一种推荐方法产生的模型或中间结果作为另一种推荐方法的输入。例如，先用基于内容的方法构建用户画像，再将用户画像作为协同过滤中相似度计算的特征维度。

本文系统采用的自适应加权融合策略同时具备加权融合与切换式混合的特征：一方面，三个子模型（内容匹配 PMFUP、兴趣漂移 PRMIDM、协同过滤 AMMBC）的得分通过加权和进行融合；另一方面，融合权重根据用户行为信号强度自适应调整——当信号稀疏时内容匹配子模型占主导，随信号积累协同过滤子模型的权重逐步提升，实现了从冷启动到个性化推荐的平滑过渡。

## 2.5 推荐系统关键问题

### 2.5.1 冷启动问题

冷启动问题（Cold Start Problem）是推荐系统面临的核心挑战之一[4]。当新用户首次使用推荐系统时，系统缺乏该用户的历史行为数据，无法通过协同过滤或行为分析来推断其偏好，导致推荐质量显著下降。冷启动问题通常分为用户冷启动（新用户无历史记录）和物品冷启动（新物品无交互记录）两类，在航空餐食推荐场景中，用户冷启动是主要挑战。

解决用户冷启动的常见策略包括：（1）利用用户注册时填写的显式偏好信息（如口味偏好、饮食禁忌）进行基于内容的推荐；（2）利用用户的人口统计学特征（如年龄、性别）与已有用户进行粗粒度匹配；（3）推荐全局热门物品作为初始推荐，随着用户行为数据的积累逐步过渡到个性化推荐；（4）采用混合推荐策略，在冷启动阶段提高内容匹配的权重，随着行为信号的增加逐步提升协同过滤的权重。本文系统综合采用了策略（1）和策略（4）：新用户首次使用时引导其完成偏好采集，并在偏好未完成时仍允许跳过直接体验推荐；推荐算法在行为信号稀疏时自动提升内容匹配子模型（PMFUP）的融合权重，随着用户交互数据的积累逐步降低其权重。

### 2.5.2 用户兴趣漂移

用户兴趣漂移（Interest Drift）是指用户的偏好和兴趣随时间发生变化的自然现象[6]。在推荐系统中，如果模型仅基于用户的长期历史行为构建静态偏好画像，而忽略了兴趣的动态变化，将导致推荐结果与用户当前的偏好产生偏差，推荐时效性和精准度下降。

处理兴趣漂移的常用方法包括：（1）时间衰减函数，即为用户的历史行为赋予随时间递减的权重，使得近期行为的贡献大于远期行为；（2）长短期偏好分离，即分别维护用户的长期稳定偏好和短期动态偏好两个模型，并在推荐时进行动态融合；（3）滑动时间窗口，即仅考虑最近一段时间内的用户行为进行偏好建模。本文系统综合采用了时间衰减函数与长短期偏好分离两种策略：行为信号通过指数衰减函数 $e^{-0.018 \times days}$ 进行时间加权（半衰期约38.5天）；同时分别维护近30天近期窗口和近180天历史窗口的两套口味偏好信号，通过对比近期偏好与历史偏好来检测口味漂移方向，对发生正向漂移的口味所对应的菜品给予得分提升。

### 2.5.3 流行度偏差

流行度偏差（Popularity Bias）是指推荐系统倾向于推荐已被大量用户选择的热门物品，而忽视了长尾物品和新物品的现象[3]。这种偏差在协同过滤中尤为明显——热门物品由于被大量用户交互过，在相似度计算和评分预测中占据优势，形成"富者愈富"的马太效应。流行度偏差不仅降低了推荐结果的多样性和个性化程度，还可能导致新物品难以获得曝光机会。

缓解流行度偏差的常用方法包括：（1）流行度惩罚因子，即在推荐得分中引入与物品流行度成反比的惩罚项，降低热门物品的得分优势；（2）反事实推理，即通过因果推断方法消除流行度对推荐结果的混杂影响；（3）去偏正则化，即在模型训练目标函数中加入流行度相关的正则化项。本文系统在协同过滤子模型（AMMBC）中引入了流行度去偏机制：对于每个候选菜品，根据其在推荐日志中的累计曝光次数计算流行度惩罚系数 $1.0 / (1.0 + \frac{\ln(1 + exposureCount)}{6.0})$，曝光次数越多，惩罚系数越低，从而在协同过滤的得分中削弱热门菜品的先发优势。同时，协同过滤子模型还引入了置信度加权策略：$clamp(simTotal / \max(1.0, neighborCount \times 0.45))$，在邻居用户数量不足或相似度总量较低时降低协同过滤得分的可信度，进一步提升推荐的稳定性。

## 2.6 推荐系统评价指标

推荐系统的评价指标用于量化评估推荐结果的质量。根据推荐任务的不同视角，评价指标通常分为预测精度指标、分类指标和排序指标三类[19]。

准确率（Accuracy）是最基础的分类指标，表示模型预测正确的样本数占总样本数的比例：

$$
Accuracy = \frac{TP + TN}{TP + TN + FP + FN}
$$

其中 $TP$（True Positive）表示正确预测为正样本的数量，$TN$（True Negative）表示正确预测为负样本的数量，$FP$（False Positive）表示错误预测为正样本的数量，$FN$（False Negative）表示错误预测为负样本的数量。在推荐场景中，正样本通常为用户实际选择或评分的物品。

精确率（Precision）衡量推荐列表中真正为用户感兴趣的物品所占的比例，反映了推荐结果的"精准度"：

$$
Precision = \frac{TP}{TP + FP}
$$

召回率（Recall）衡量用户真正感兴趣的所有物品中有多大比例被推荐系统成功找出，反映了推荐结果的"查全率"：

$$
Recall = \frac{TP}{TP + FN}
$$

F1分数（F1-Score）是精确率与召回率的调和平均数，用于在两个指标之间取得平衡：

$$
F1 = 2 \times \frac{Precision \times Recall}{Precision + Recall}
$$

AUC（Area Under the ROC Curve）是评估推荐模型排序质量的重要指标。ROC曲线以假正率（FPR）为横轴、真正率（TPR）为纵轴，通过改变分类阈值绘制，AUC值即为ROC曲线下的面积，取值范围为 $[0.5, 1]$。AUC值越大，表示模型区分正负样本的能力越强：$AUC = 0.5$ 表示模型预测能力等同于随机猜测，$AUC = 1$ 表示模型能完美区分所有正负样本。

## 2.7 系统开发技术选型

### 2.7.1 后端技术：Spring Boot 3 与 MyBatis

本系统后端采用 Spring Boot 3.2.5 作为核心开发框架，基于 JDK 17 运行环境。Spring Boot 是 Spring 生态体系的快速开发框架，通过自动配置（Auto-Configuration）和内嵌服务器（Embedded Server）大幅简化了 Spring 应用的搭建与部署流程。其主要优势包括：起步依赖（Starter）机制将相关依赖打包为单一引用，减少了依赖管理的复杂性；内嵌 Tomcat 服务器使应用可以以独立 JAR 包形式运行，无需外部 Servlet 容器；与 Spring Security、Spring Data 等 Spring 生态组件的无缝集成能力。

数据持久层采用 MyBatis 3.0.3 作为 ORM 框架。MyBatis 是一种半自动化的持久层框架，通过 XML 映射文件或注解将 Java 对象与 SQL 语句关联，开发者可以直接编写和优化 SQL 语句，在保持 SQL 灵活性的同时避免了 JDBC 的繁琐编码。本系统通过 application.yml 配置文件指定了 Mapper 文件的扫描路径（`classpath:mapper/*.xml`）、实体类别名包（`fun.hykgraph.entity`）以及驼峰命名自动映射（`map-underscore-to-camel-case: true`），共包含11个 Mapper XML 映射文件和对应的 Java Mapper 接口，覆盖用户、航班、菜品、偏好、推荐日志、预选订单与评分等核心数据表。

### 2.7.2 前端技术：Vue3、Element Plus 与 UniApp

管理端前端采用 Vue 3.4.21 作为核心框架，配合 TypeScript 5.4.0 进行类型安全的开发，使用 Vite 5.2.8 作为构建工具。Vue 3 的组合式 API（Composition API）允许将组件逻辑按功能进行模块化组织，提升了代码的可复用性和可维护性。UI 组件库采用 Element Plus 2.7.1，提供表格、表单、对话框、导航菜单等丰富的企业级组件。数据可视化采用 ECharts 5.5.0，用于管理端数据看板的图表展示。状态管理采用 Pinia 2.1.7，配合持久化插件（pinia-plugin-persistedstate 3.2.1）将关键状态自动同步至浏览器本地存储。HTTP 客户端采用 Axios 1.6.8，用于向后端发送 RESTful API 请求。

小程序端采用 UniApp 3.0.0 跨端开发框架，基于 Vue 3.3.11 构建，编译目标为微信小程序（mp-weixin）。UniApp 通过统一的代码基输出到多端平台，降低了多端维护成本。小程序端集成了微信一键登录功能，通过调用 `wx.login` 获取临时凭证后向后端请求 OpenID 并生成 JWT 令牌，后续请求通过 HTTP Header 携带令牌完成身份认证。

### 2.7.3 数据存储：MySQL 与 Redis

本系统采用 MySQL 作为主数据库，数据库名为 `aviation_food_recommend`，运行于本地 3306 端口。MySQL 是一种成熟的关系型数据库管理系统，支持 ACID 事务特性，适合存储具有强关联关系的结构化数据。本系统的核心数据表包括用户表（user）、航班信息表（flight_info）、菜品表（dish）、用户偏好表（user_preference）、推荐日志表（recommendation_log）、餐食预选表（meal_selection）、评分任务表（flight_service_rating）和公告表（flight_announcement）等，表之间通过外键和索引建立关联约束。

本系统采用 Redis 作为缓存数据库，运行于本地 6379 端口（db 1）。Redis 是一种基于内存的高性能键值存储数据库，支持字符串、哈希、列表、集合和有序集合等多种数据结构。在本系统中，Redis 主要用于缓存热点数据（如航班列表、菜品信息），以降低数据库查询压力，提升接口响应速度。

## 2.8 本章小结

本章围绕航空餐食智能推荐系统所涉及的基础理论与关键技术进行了系统介绍。首先从推荐系统的概念与分类入手，阐述了三类主要推荐方法的原理与特点；其次深入介绍了协同过滤推荐的核心机制——基于用户的协同过滤、余弦相似度计算以及收缩因子修正策略；然后分别介绍了基于内容的推荐方法和混合推荐策略；在此基础上，分析了推荐系统面临的三类关键问题——冷启动、兴趣漂移与流行度偏差，并给出了相应的解决思路；接着介绍了推荐系统常用的评价指标及其计算方法；最后概述了本系统实现所采用的技术栈，包括后端框架 Spring Boot 3 与 MyBatis、前端框架 Vue3 与 UniApp，以及 MySQL 与 Redis 数据存储方案。本章为后续第三章推荐算法的详细设计和第四章系统实现提供了必要的理论和技术基础。
