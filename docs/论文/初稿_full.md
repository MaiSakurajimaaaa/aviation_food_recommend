**摘 要**

随着民航业快速复苏，机上餐食服务成为影响旅客满意度的关键环节。传统标准化配餐模式难以满足旅客口味偏好与饮食需求的个性化差异，导致餐食浪费与旅客体验不佳并存。基于此背景，本文设计并实现了一个面向航空旅客的智能美食推荐系统，形成\"偏好采集---智能推荐---预选确认---航后评分\"的完整闭环。

本文设计了一套融合用户画像匹配（PMFUP）、兴趣漂移检测（PRMIDM）与修正协同过滤（AMMBC）的混合推荐方案。PMFUP 基于用户显式偏好进行内容匹配；PRMIDM 通过对比近 30 天与近 180 天行为信号检测口味漂移；AMMBC 在协同过滤中引入收缩因子（λ=3.0）、流行度去偏与置信度加权。三子模型由自适应权重解析器动态融合，并引入舱位等级递进规则与前序餐食惩罚机制以适配航空场景。系统基于 Spring Boot 3 + Vue3 + UniApp 技术栈实现管理端、小程序端和后端三端架构。

实验设计了两组对照数据集：数据集 A 为 86 人、10 道菜品的同质模拟数据，数据集 B 为 188 人、30 道菜品的异构模拟数据（含口味漂移组、偏好邻居组与冷启动组以模拟真实场景的信号异质性）。采用亚采样与 Bootstrap 重采样方法对比自适应融合、固定权重融合、纯内容匹配和纯协同过滤四种策略。核心发现如下：（1）在同质数据上，自适应权重与固定权重 \[0.60, 0.25, 0.15\] 的行为完全等价------亚采样（45/60/86 人）与 Bootstrap 重采样（200/500/1000 人）中两者的所有指标差距均为 0.0%；（2）在异构数据上，在本文构造的异构模拟数据集上，自适应融合相对于固定权重表现出正向增益------Bootstrap 重采样至 200 人时 Top-3 召回率领先 17.0 个百分点（19.3% vs 2.3%），至 1000 人时领先 4.2 个百分点（16.1% vs 11.9%），500 人规模下自适应融合首次在 Top-1 准确率上取得突破（2.0% vs 0.0%）；（3）纯 PMFUP 在候选池标签区分度较高时具有结构性优势（数据集 A Top-1 28.6%），但此优势在数据集 B 的 30 道菜品（标签高度重叠）上完全消失------固定权重的 Top-3 降至 0%，仅自适应融合仍能维持有效推荐。候选池标签相似度分析进一步表明，基于标签的内容匹配存在一个由标签空间容量决定的结构性性能上限，当候选菜品扩展至真实场景规模时，行为驱动的漂移检测与协同过滤将成为突破此上限的必要手段。

**关键词**：推荐系统；航空餐食；混合推荐；协同过滤；兴趣漂移；Spring Boot

**Abstract**

With the rapid recovery of civil aviation, in-flight meal service has become key to passenger satisfaction. This paper designs an intelligent food recommendation system for aviation passengers, featuring a hybrid recommendation scheme that integrates content-based profile matching (PMFUP), short-term taste drift detection (PRMIDM), and refined collaborative filtering (AMMBC) with shrinkage, popularity debiasing, and confidence weighting. Three sub-models are fused via an adaptive weight resolver, with cabin-class tiered visibility rules and sequential meal penalty mechanisms.

Experiments are conducted on two datasets: Dataset A (86 users, 10 dishes, homogeneous) and Dataset B (188 users, 30 dishes, heterogeneous, with taste-drift groups, preference-neighbor clusters, and cold-start users). Key findings: (1) On homogeneous data, adaptive weights are behaviorally equivalent to fixed weights \[0.60, 0.25, 0.15\] across all scales (0.0% gap under subsampling and bootstrap); (2) On heterogeneous data, adaptive fusion consistently outperforms fixed weights---Top-3 recall leads by 17.0 pp at 200 bootstrap users (19.3% vs 2.3%), narrowing to 4.2 pp at 1,000 users (16.1% vs 11.9%); at 500 users, adaptive fusion achieves its first Top-1 accuracy at 2.0% (vs 0.0% for fixed); (3) Pure PMFUP\'s structural advantage on sparse but well-differentiated tags (28.6% Top-1 on Dataset A) vanishes when tag overlap increases on Dataset B, where only adaptive fusion maintains effective recommendation. Tag similarity analysis reveals a structural performance ceiling for content-based matching determined by tag space capacity.

**Keywords**: Recommendation System; Aviation Catering; Hybrid Recommendation; Collaborative Filtering; Interest Drift; Spring Boot

# 第1章引言

## 1.1 研究背景及意义

随着全球民用航空业的持续复苏，机上餐食服务已从传统\"配餐保障\"转变为影响旅客满意度的关键服务触点。2024年我国民航完成旅客运输量7.3亿人次，全国航空配餐企业年配餐量近7亿份。在如此庞大的服务规模下，机上餐食的品质与个性化程度直接影响航空公司的品牌竞争力与运营效率。

然而，传统的机上配餐模式长期沿用\"标准化、批量供应\"策略，旅客在值机时面对有限的两到三种餐食选项，无法根据个人口味偏好、饮食禁忌或健康需求进行自主选择。这一模式面临三个核心矛盾：其一，旅客饮食需求的多样化与餐食供给的标准化之间的矛盾------不同旅客在口味、餐食类型和健康偏好上存在显著差异，标准化配餐难以覆盖长尾需求；其二，航空公司的成本控制与餐食浪费之间的矛盾------全球航空业每年产生约360万吨客舱和餐食垃圾，其中18%至20%为未食用餐食，过度装载导致的浪费直接转化为燃油消耗与碳排放的增加；其三，旅客期望的实时个性化服务与现有配餐流程信息滞后之间的矛盾------旅客偏好数据在订票、值机、登机各环节碎片化分布，缺乏统一的采集与智能推荐机制。

从理论层面看，个性化推荐系统在电子商务、社交媒体等领域已取得丰硕成果，协同过滤、基于内容的推荐与混合推荐等方法日趋成熟^\[1\]\[2\]^。然而，将通用推荐技术迁移到航空餐食这一垂直场景时，面临一系列特有约束：航班餐食供给受航线、舱位与库存的多维约束，推荐结果必须在实际可供应范围内；旅客餐食偏好可能随季节、航线等因素发生短期漂移，静态偏好模型难以捕捉动态变化^\[3\]^；新旅客首次乘机缺乏历史行为数据，构成典型的冷启动问题^\[4\]^；多餐食航班上需考虑同一旅客前序餐次的选择，避免重复推荐，本文引入前序餐食惩罚机制。上述约束使得航空餐食推荐成为兼具理论价值与应用价值的研究问题。

从应用层面看，构建一个面向航空旅客的智能美食推荐系统具有多重现实意义：对旅客而言，个性化推荐能提升服务体验，满足多元化饮食需求；对航空公司而言，基于偏好预测的精准配餐能减少餐食浪费、降低运营成本；对行业而言，该系统可为铁路、邮轮等其他交通领域的餐饮服务数字化提供参考范式。

基于此背景，设计了一套融合用户画像匹配、兴趣漂移建模与协同过滤的混合推荐方案，将其工程落地为管理端、小程序端与后端服务三端系统，并基于Spring Boot + Vue.js + UniApp技术栈构建了管理端、小程序端与后端服务三端系统，形成\"偏好采集---智能推荐---预选确认---航后评分\"的完整闭环。

## 1.2 国内外发展现状与发展趋势

### 1.2.1 国内研究概况

国内推荐系统的研究起步虽晚于国外，但近年来在算法改进与行业应用方面均取得显著进展。在协同过滤与混合推荐方面，余江龙等人从多维度综述了个性化推荐系统的研究方法^\[1\]^；王淼等人将基于内容的推荐与协同过滤算法相结合，构建混合推荐系统，改善了数据稀疏性问题^\[2\]^。在兴趣漂移与动态偏好方面，夏翔等人在相似度计算中引入时间权重与兴趣迁移模型^\[3\]^；钱聪等人融合时间衰减函数与评论语义相似度，捕捉用户兴趣的渐变性变化^\[6\]^；刘超等人通过提取长短期偏好并自适应聚合来处理序列推荐中的偏好漂移^\[5\]^，为本文的多餐时序建模提供了参考。在冷启动问题上，毛骞等人按数据驱动与方法驱动两类进行了系统综述^\[4\]^。在协同过滤相似度优化方面，刘昊东等人在余弦相似度中引入Jaccard函数与热门度修正因子^\[7\]^，并结合置信度加权策略降低流行偏差对推荐结果的影响。

在餐饮推荐领域，张明颜利用本体构建与情境感知技术，对移动餐饮平台的情境要素进行建模，实现个性化餐饮推荐^\[9\]^；王佳安等人设计并实现美食精准搜索与智能推荐平台[^\[10\]^]{.smallcaps}；李凯提出了一种融合项目特征与用户偏好的美食推荐方法^\[11\]^，为本文提供了直接参考。在航空餐食信息化建设方面，北京航食于2024年上线了旅客餐食预订管理系统，贯通19套系统实现了全链条数字化管理。厦门航空率先推出经济舱免费选餐服务，东航申请了航班配餐量智能预测专利。但将个性化推荐算法深度嵌入航空餐食场景、形成完整闭环系统的工程实践仍较为缺乏。

### 1.2.2 国外研究概况

国外推荐系统研究起步较早，理论体系更为成熟。在基础理论方面，Koren等人提出的矩阵分解技术\[16\]通过学习用户与物品的隐语义表示，显著提升了推荐精度与可扩展性，为协同过滤奠定了数学基础。在食品推荐领域，Bondevik等人对食品推荐系统进行了系统性综述\[17\]，指出多数系统依赖显式偏好与物品属性特征进行匹配，而将行为反馈融入推荐闭环的研究仍较为薄弱。

在航空餐食智能化方面，Dwarampudi（2025）提出了基于AI的机上餐食个性化框架\[18\]，比较了协同过滤、旅客细分、强化学习和神经网络四条技术路径。在产业实践方面，汉莎航空推出\"餐盘追踪器\"系统，通过AI视觉识别回收餐食的剩余状态以优化配餐量；荷兰皇家航空的TRAYS模型据报可减少63%的餐食浪费；空客公司于2025年推出\"Smart Catering\"解决方案，通过机载摄像头与云端偏好分析实现智能配餐。总体来看，国外研究更注重端到端的数据闭环，但仍面临真实数据稀缺、个性化与成本平衡等挑战。

### 1.2.3 航空餐食推荐发展历程

航空餐食服务模式大致经历了四个发展阶段。第一阶段（20世纪50年代至90年代）为标准化配餐阶段，机上餐食作为基础飞行保障服务批量供应，不考虑个性化需求。

第二阶段（20世纪90年代末至2010年代）为特殊餐食预申请阶段，航空公司开始在购票环节提供清真餐、素食餐等特殊选项，旅客需提前24至48小时申请，但普通旅客仍仅有标准餐可选。

第三阶段（2010年代至2020年代初）为数字化预选阶段，航空公司通过App或网站提供餐食预选服务，但推荐逻辑以基于规则的简单排序为主，尚未实现个性化推荐。

第四阶段（2020年至今）为智能推荐探索阶段，航空公司开始利用机器学习算法分析旅客偏好与行为信号，AI视觉识别技术被引入餐盘回收环节，形成从\"推荐→消费→反馈\"的数据闭环，这一阶段的核心特征是推荐系统、需求预测与数据闭环的初步融合，也是本文工作的技术背景。

### 1.2.4 航空餐食推荐发展趋势

综合国内外研究现状与航空餐食服务模式的演进，该领域呈现以下发展趋势。（1）从简单规则到混合算法。早期主要基于销量排序等简单规则，未来将向多模型融合的混合推荐算法演进，协同过滤、内容匹配与行为反馈的互补融合将显著提升推荐精度\[2\]。（2）从静态画像到动态偏好。传统的静态偏好采集正在被时间衰减函数与兴趣漂移检测技术所替代\[3\]\[6\]，通过对比近期行为与长期历史实现自适应偏好更新。（3）从单向推荐到闭环反馈。在推荐曝光、点击浏览、预选确认、航后评分等环节埋设行为追踪点，构建从\"曝光→交互→选择→评价\"的数据闭环，使推荐算法持续迭代优化\[11\]。（4）从单次推荐到序列推荐。多餐食航班上需考虑同一旅客前序餐次的选择，避免重复推荐，序列推荐方法将多餐推荐建模为时序决策问题\[5\]。（5）从同质化到多层次差异化。引入舱位等级递进的餐食可见规则与差异化推荐策略，实现头等舱、商务舱与经济舱旅客的精准服务。（6）从数据孤岛到全链路数字化。当前旅客偏好数据在订票、值机、配餐、客舱等系统间分散存储，构建统一的偏好数据平台是全链路数字化的关键方向。

## 1.3 研究内容及结构

本文的研究内容与贡献定位于两个层面。在系统工程层面，针对航空餐食推荐场景中供给约束强、偏好动态变化、交互信号稀疏的特征，完成了领域建模（12 张数据表，涵盖航线-菜品绑定、舱位等级递进、多餐时序约束和库存限额），设计并实现了集成上述约束的管理端、小程序端和后端服务三端系统（70 个 REST API 端点，两个定时任务），形成从偏好采集到航后评分的完整业务闭环。在算法分析层面，设计了一套融合内容匹配、兴趣漂移检测与修正协同过滤的混合推荐方案，并通过亚采样分析、Bootstrap 重采样、候选池特征分析和消融对比，系统性地揭示了不同推荐策略在小规模、同质化数据上的性能边界------特别是内容匹配在标签区分度较高时的结构性优势、协同过滤在用户规模不足时的失效原因、以及自适应权重机制在数据未达临界规模前的行为等价性。这些发现为同类数据稀疏场景下的推荐策略选型提供了经验性指导，并为未来在更大规模真实数据上的算法验证画出了清晰的路线图。各章节安排如下：

# 第 2 章 相关基础知识

## 2.1 推荐系统概述

[]{#_Toc228979259 .anchor}推荐系统是一种解决信息过载问题的信息过滤技术，其核心任务是根据用户的历史行为、个人偏好以及物品的属性特征，从海量候选物品中筛选出用户可能感兴趣的物品子集，并以个性化列表的形式呈现。与搜索引擎不同，推荐系统无需用户主动输入查询关键词，而是通过分析隐式行为（浏览、点击、选择）和显式反馈（评分、点赞）自动推断用户兴趣偏好。

根据推荐策略的不同，主流推荐方法可分为三类：

（1）协同过滤推荐：基于"相似用户具有相似偏好"的假设，利用用户群体的行为数据进行推荐，不依赖物品的内容特征，具有良好的领域适用性。

（2）基于内容的推荐：从物品自身的属性特征出发，通过匹配用户兴趣画像与物品特征生成推荐，不依赖其他用户数据，不存在物品冷启动问题。

（3）混合推荐：将多种推荐方法以加权融合、切换式混合、特征组合或分层混合等策略进行组合，以弥补单一方法的不足。本文设计的方案即属于自适应加权融合策略的范畴。

## 2.2 协同过滤

[]{#_Toc228979264 .anchor}基于用户的协同过滤是协同过滤中最经典的方法之一\[7\]，其核心步骤为：计算目标用户与其他用户的相似度，选取最相近的K个邻居，再根据邻居对物品的行为加权预测目标用户的偏好。该方法能够发现用户潜在兴趣并具备一定惊喜度，但在用户规模庞大时计算复杂度较高，且面临新用户冷启动问题。

余弦相似度是计算用户间相似度最常用的方法之一\[8\]。针对稀疏数据下小重叠度可能产生假性高相似的问题，本文引入收缩因子对相似度进行修正\[12\]：当两个用户的共同交互物品数量较少时，对相似度施加惩罚使其向零收缩，仅在拥有充分共同交互证据时才认定为高度相似。此外，本文还引入流行度去偏与置信度加权机制，以缓解热门菜品的马太效应并在邻居证据不足时降低协同过滤得分的可信度。具体公式与实现细节见第四章AMMBC子模型一节。

## 2.3基于内容的推荐

基于内容的推荐从物品属性出发，通过构建用户兴趣画像并与物品特征进行匹配来生成推荐^\[13\]^。本系统中，每道菜品具有餐食类型（儿童餐/标准餐/清真餐/素食）和口味标签（清淡、咸香、微辣、甜口、低脂、高蛋白）两类内容特征；用户画像来源于显式偏好设置与历史行为累积。该方法的主要优势在于不存在物品冷启动问题且推荐可解释性强，局限在于难以挖掘潜在兴趣且可能缺乏多样性。具体匹配度计算方式见第四章PMFUP子模型一节。

## 2.4 混合推荐方法

混合推荐方法通过将多种推荐策略以一定方式进行组合，取长补短，以克服单一推荐方法的局限性^\[2\]^。[]{#_Toc228979266 .anchor}本文采用自适应加权融合策略：三个子模型（内容匹配PMFUP、兴趣漂移PRMIDM、协同过滤AMMBC）的得分通过加权和融合，而权重根据用户行为信号强度自适应调整------信号稀疏时内容匹配占主导，随信号积累逐步提升协同过滤权重，实现从冷启动到个性化推荐的平滑过渡。

## 2.5推荐系统关键问题

[]{#_Toc228979270 .anchor}冷启动问题：新用户缺乏历史行为数据，导致推荐质量下降。本文通过引导用户完成偏好采集，并在算法层面于信号稀疏时自动提升内容匹配子模型的融合权重来解决。

用户兴趣漂移：用户偏好随时间动态变化，静态画像难以捕捉。本文综合采用时间衰减函数与长短期偏好分离策略，对比近30天近期窗口与近180天历史窗口的行为信号，检测口味漂移方向。

流行度偏差：协同过滤倾向于推荐热门物品，形成马太效应。本文在协同过滤子模型中引入流行度惩罚系数与置信度加权策略予以缓解，具体实现见第四章。

## **2.6本章小结**

本章阐述了推荐系统的基本概念与三类主要方法，分别介绍了协同过滤、基于内容的推荐及混合推荐策略的核心思想，分析了冷启动、兴趣漂移和流行度偏差三类关键问题并给出了本文的解决思路。具体算法公式与实现细节将在第四章中结合系统设计逐项展开。

#  第3章 需求分析

## 3.1 系统需求概述

航空餐食推荐系统需要在航班供给约束、旅客个性化偏好和航空公司运营效率三者之间取得平衡。传统配餐模式下，旅客在值机时面对有限选项，无法根据口味偏好自主选择；航空公司按座位数估算配餐量，缺乏精准的需求预测手段，导致餐食浪费与旅客满意度低下并存。

本系统的核心目标是在航班餐食供给的硬约束下，为每位旅客生成个性化的餐食推荐列表，并提供从偏好采集、餐食预选到航后评分的完整服务闭环。系统需同时服务两类用户角色：航空公司运营管理人员通过管理端进行航班调度、餐食管理和数据监控；航空旅客通过小程序端完成航班绑定、口味偏好设置、餐食预选和航后评分。

## 3.2 系统角色分析

本系统涉及两类用户角色，其职责与需求边界明确。

（1）**管理员**：管理员是航空公司运营人员，通过PC端Web管理界面操作系统。其核心职责包括：维护航班信息（航班号、起降地、起降时间、餐食数量与预选截止时间）；管理菜品资源（菜品名称、餐食类型、口味标签、库存）；为航班-航线配置可选的菜品绑定关系并按舱位类型差异化设置；查看推荐数据看板（推荐次数、评分均值、热门菜品分布）；监控评分任务状态并对过期或异常任务进行重开或手动过期；发布航班公告（如选餐开放通知、截止提醒）；维护旅客信息（舱位类型设定、航班-旅客关联）。

（2）**旅客**：旅客是航空公司的乘客，通过微信小程序使用系统。其核心操作流程为：首次使用时通过微信授权登录，输入身份证号完成身份核验；浏览并绑定与自己身份证号关联的候选航班；设置个人的餐食类型偏好（儿童餐/标准餐/清真餐/素食）和口味偏好（清淡、咸香、微辣、甜口、低脂、高蛋白六种可选）；在航班餐食推荐列表中浏览菜品并完成预选确认，支持对已选餐食的改选操作；航班结束后对已选菜品进行1至5星的航后评分，支持暂缓评价功能。

两角色的核心差异在于：管理员面向全部旅客和航班的批量管理，关注运营效率和数据宏观趋势；旅客面向个人偏好和个性化服务，关注推荐精准度和操作便捷性。两角色通过系统数据库实现数据共享------管理员配置的航班-菜品绑定决定了旅客可见的候选菜品范围，旅客的预选和评分数据回流至推荐算法和看板统计，形成完整的运营闭环。该系统的角色用例图如图3.1所示。

##  3.3 系统功能需求

###  3.3.1 管理端功能需求

管理端面向航空公司运营人员，需实现航班管理、餐食管理、航线配餐、数据监控和人员管理五类功能。

（1）航班管理。管理员能够创建、编辑、查询和删除航班信息。航班信息包括航班号、起降地、起降时间、飞行时长、餐食数量与餐食时段（如午餐、加餐）、预选截止时间。系统需自动根据起降时间计算飞行时长，并根据飞行时长（≥180分钟则含两餐）自动设置餐食数量。航班列表支持按航班号搜索，支持按状态筛选（启用/禁用）。

（2）餐食管理。管理员能够创建、编辑、查询和上架/下架菜品。菜品信息包括菜品名称、图片、描述文字、所属分类、餐食类型（儿童餐/标准餐/清真餐/素食四选一）、口味标签（清淡、咸香、微辣、甜口、低脂、高蛋白六种多选）和库存数量。菜品分类独立管理，支持按分类筛选菜品。

（3）航线配餐。管理员能够为每个航班-航线配置可选的菜品绑定关系，并指定每种绑定的舱位类型（头等舱/商务舱/经济舱）。系统需按舱位等级递进规则进行可见性过滤：头等舱旅客可见全部三个舱位的菜品，商务舱旅客可见商务舱和经济舱菜品，经济舱旅客仅见经济舱菜品。该规则同时支持管理端配餐操作和用户端推荐结果的舱位过滤。

（4）数据监控。管理端首页以数据看板形式展示推荐系统的核心运营指标，包括推荐总次数、评分均值、热门菜品分布和异常用户数量。评分管理中心以表格形式列出所有航后评分任务，支持按评分状态（待评分/已提交/已暂缓/已过期）和航班号筛选，支持对异常任务的重开或手动过期操作。用户餐食中心展示旅客的预选记录，支持按航班号统计各菜品的预选需求量分布。

（5）人员管理。超级管理员能够创建、编辑、查询和启用/禁用管理端员工账号。员工登录时需通过账号和密码认证，系统签发JWT令牌用于后续请求的身份验证。

### 3.3.2 小程序端功能需求

小程序端面向航空旅客，需实现用户登录、航班绑定、偏好采集、餐食预选和航后评分五类功能，构成完整的旅客使用闭环。

（1）微信登录与身份核验。旅客首次启动小程序时，通过微信一键授权登录。系统通过微信API获取用户OpenID并完成账号注册或登录，签发JWT令牌用于后续认证。登录后旅客需输入18位身份证号完成身份核验，前端应进行基本格式校验。身份核验后系统展示旅客的舱位类型（由管理员在后台设定）及相关权限说明。

（2）航班绑定。系统根据旅客的身份证号匹配该旅客所关联的候选航班列表，以卡片形式展示航班号、起降地、起飞时间和预选截止时间。旅客选择一个航班完成绑定。已绑定的航班信息持久化显示于首页。

（3）偏好采集。首次使用或首次进入推荐前，系统引导旅客设置餐食偏好。偏好设置包含餐食类型（四选一，可留空跳过）和口味标签（六种多选，可留空跳过）两类选项。旅客可选择\"保存并进入推荐\"提交偏好，也可\"先跳过，直接体验推荐\"以未设偏好状态进入推荐------此时推荐算法使用默认权重和基础内容匹配兜底。

（4）餐食预选与改选。系统根据旅客所绑定航班的航线-菜品配置和舱位类型，通过推荐算法生成个性化菜品推荐列表并展示。每张菜品卡片展示菜品图片、名称、得分百分比、餐食类型、口味标签和推荐理由。旅客点击菜品进入确认页，确认后完成预选并扣减菜品库存。已预选的餐次支持改选操作------重新选择后旧预选记录被更新，库存同步调整。多餐食航班（如含午餐和加餐）中需展示餐次选择器，旅客可逐餐完成预选，系统对前序餐食在后续餐次中施加多样性惩罚（相同菜品得分降权30%、同类菜品降权15%）。预选截止时间后操作按钮自动禁用。

（5）航后评分。航班落地后系统自动生成评分任务。旅客进入评分列表，可查看待评价的航班和已选菜品，以五星评分组件进行1至5分评价。评分提交后任务状态更新为已提交。旅客也可选择暂缓评价，24小时后重新提醒。若7天内未提交评分则自动过期，管理员可通过后台重开过期任务。

本章从系统需求概述、角色分析和功能需求三个层面对航空旅客智能美食推荐系统进行了系统化需求分析。明确了管理员和旅客两类角色的职责边界与使用流程，按管理端和小程序端两个维度详细阐述了航班管理、餐食管理、航线配餐、数据监控、人员管理以及微信登录、航班绑定、偏好采集、餐食预选和航后评分等功能需求。需求分析的结论为第四章系统设计和第五章系统验证提供了明确的验收依据。

###  4.4.1 算法总体设计

如第 1 章所述，航空餐食推荐面临航线绑定、偏好漂移与信号稀疏三重约束。本章详述 Fused-PMFUP-PRMIDM-AMMBC 混合推荐方案的设计细节。算法以一组静态常量集中管理核心参数，包括允许的餐食类型集合 {1, 2, 3, 4}（依次对应儿童餐、标准餐、清真餐、素食）、允许的口味标签集合 {\"清淡\", \"咸香\", \"微辣\", \"甜口\", \"低脂\", \"高蛋白\"}、舱位等级常量（CABIN_FIRST=1, CABIN_BUSINESS=2, CABIN_ECONOMY=3）、协同过滤收缩系数 CF_SHRINK_LAMBDA=3.0、近期窗口 RECENT_WINDOW_DAYS=30、历史窗口 HISTORY_WINDOW_DAYS=180 以及曝光信号基础权重 EXPOSURE_SCORE_WEIGHT=0.15。舱位等级递进规则 resolveCabinCascade 的实现逻辑为：若用户舱位为头等舱（1），返回可见类型 \[1, 2, 3\]；若为商务舱（2），返回 \[2, 3\]；若为经济舱（3）或空值，返回 \[3\]。该规则确保高舱位旅客可以浏览低舱位的餐食选项，反之则不可见。候选菜品通过 SQL 查询从 flight_route_dish 航线绑定表中筛选，限定条件为航线匹配、舱位等级可见（cabin_type IN (\...) 递进规则）、库存大于零，单次最多返回20条。算法架构如图3.1所示。

###  4.4.2 PMFUP子模型

PMFUP 子模型利用用户显式设定的偏好（餐食类型、口味标签）与菜品属性进行内容匹配，在行为信号稀疏时提供主要的推荐依据。菜品包含两种内容特征：mealType（1=儿童餐、2=标准餐、3=清真餐、4=素食）和 flavorTags（清淡、咸香、微辣、甜口、低脂、高蛋白中的若干标签）。

评分计算分三步：基础分统一为0.2；若用户偏好集中的餐食类型与菜品匹配，追加奖励------单偏好时追加0.45（明确偏好），多偏好时追加0.38（宽泛偏好）；若口味标签匹配，设匹配数为 $n$，口味奖励为 $\backslash min(0.35,0.15 + n\backslash times0.10)$。最终得分钳位至 $\lbrack 0,1\rbrack$：

$$S_{pmfup} = clamp(0.2 + typeBonus + flavorBonus)$$

基础分 0.2 保证了无偏好匹配的菜品不会得分为零，使其仍有机会通过其他子模型获得推荐。餐型匹配中单偏好（+0.45）与多偏好（+0.38）的差异化反映了偏好明确性的影响------用户仅设定一种偏好类型时匹配意图更聚焦。

PMFUP 得分不依赖航线、时间与行为关联数据，保证了冷启动场景下的可用性。其本质是用户显式声明的偏好与菜品属性标签之间的集合匹配，匹配粒度越细（口味标签重合越多），得分越高，但受上限0.35约束以避免口味匹配过度主导总分。餐食类型匹配的奖励差异化（0.45 vs 0.38）则反映了偏好明确程度的区分：单一偏好意味着用户决策聚焦，匹配应获更高权重。

###  4.4.3 PRMIDM子模型

30天与180天双窗口的选取基于以下考量：30天通常覆盖1-2个航旅周期内的饮食选择，反映短期漂移趋势；180 天积累半年的行为记录，作为稳定的口味基线。两者的差值（driftBoost）仅在近期偏好显著高于历史偏好时为正，避免了将\"用户一贯喜欢某口味\"误判为漂移。

RMIDM 子模型通过对比近期（30天）与历史（180天）的口味偏好信号，检测用户口味的短期漂移方向。对于候选菜品 $d$，计算步骤如下。

（1）从交互上下文的 userRecentFlavorScore 和 userLongFlavorScore 中，汇总用户对菜品所含各口味的近期得分 $R_{f}$ 与历史得分 $H_{f}$：

$$recentScore = \sum_{f}^{}R_{f},\quad historyScore = \sum_{f}^{}H_{f}$$

（2）采用 sigmoid 型归一化函数将两类得分映射至 $\lbrack 0,1\rbrack$。归一化函数的选择需满足三个条件：在 $x = 0$ 附近近似线性以保持小信号的区分度，在 $x$ 较大时趋于饱和以避免少数极端值过度支配，值域为 $\lbrack 0,1)$。本文采用的归一化公式为：

$$norm(x) = \frac{x}{x + 2.0}$$

其实现代码为 private double normalizeSignal(double score) { return clamp01(score / (score + 2.0)); }，其中常数2.0控制饱和速度：$x = 2$ 时归一化值为0.5，$x = 8$ 时约为0.8。将两类得分分别归一化：

$$recentPref = norm(recentScore),\quad historyPref = norm(historyScore)$$

（3）漂移增量定义为近期偏好减去历史偏好的正向差值：$driftBoost = clamp(recentPref - historyPref)$，仅当近期偏好显著高于历史时才为正。同时获取用户对该菜品的直接兴趣得分 $interest$，计算短期菜品信号 $dishSignal = clamp\left( interest\text{/}8.0 \right)$。

（4）加权合成：30天与180天双窗口的选取基于航旅周期特征：30天覆盖1-2次出行周期内的饮食选择，反映短期漂移趋势；180天积累约半年的行为记录，代表稳定的长期口味基线。

$$S_{prmidm} = clamp(0.60 \times recentPref + 0.30 \times driftBoost + 0.10 \times dishSignal)$$

###  4.4.4 AMMBC子模型

AMMBC 子模型采用基于用户的协同过滤，在标准余弦相似度的基础上引入三种修正机制：收缩因子修正小重叠度下的假性高相似，流行度去偏缓解热门菜品的马太效应，置信度加权在邻居证据不足时降低预测可信度。余弦相似度采用单遍点积方式实现，适用于稀疏的用户-菜品得分向量------遍历第一个向量的键集同时累加点积和第一个范数，再单独累加第二个向量的范数，最后以点积除以两个范数平方根的乘积。算法对每个有效邻居（行为向量中包含目标菜品的其他用户）执行以下步骤：计算余弦相似度；统计两用户行为向量的共同键数量作为重叠度；应用收缩因子修正相似度；将修正相似度乘以邻居对该菜品的得分，累加到加权预测值中。

收缩因子修正：设两个用户共同交互的菜品数量为 $overlap$，收缩系数 $\lambda = \ 3.0$，修正后的相似度为：

$$adjustedSim = sim \times \frac{overlap}{overlap + 3.0}$$

该修正的效果可通过数值说明：$overlap = 1$ 时修正系数为0.25，相似度被大幅削弱，避免因偶然的单一共同菜品产生假性高相似；$overlap = 3$ 时修正系数为0.5，相似度折半保留；$overlap = 10$ 时修正系数约为0.77，逐步逼近原始相似度。该机制使得两个用户只有在拥有充分共同交互证据时才被认定为高度相似。

流行度去偏：设菜品累计曝光次数为 $\exp$，惩罚系数为：

$$popDebias = \frac{1.0}{1.0 + \frac{\ln(1 + exp)}{6.0}}$$

置信度加权：设邻居相似度总量为 $simTotal$，邻居数为 $neighborCount$，置信度定义为：

$$confidence = clamp\left( \frac{simTotal}{\max(1.0,neighborCount \times 0.45)} \right)$$

该公式的含义为：当每个邻居的平均修正相似度达到0.45时置信度为1.0；若邻居的平均相似度偏低（如多为0.2左右的弱关联用户），置信度按比例降低。置信度综合反映了\"邻居群体在统计意义上是否足够可靠\"。例如，3个邻居每人平均相似度仅0.2，则 $simTotal = 0.6$，$confidence = 0.6\text{/}\max(1.0,3 \times 0.45) = 0.6\text{/}1.35 \approx 0.44$，协同项的权重系数 $0.70 + 0.30 \times 0.44 = 0.83$，协同得分被适度降低。

AMMBC 最终得分由协同预测分与三种修正因子合成：

$$S_{ammbc} = clamp\left( collaborative \times popDebias \times (0.70 + 0.30 \times confidence) \right)$$

其中 $collaborative = clamp\left( \frac{simWeighted\text{/}simTotal}{8.0} \right)$，系数8.0为归一化缩放因子------当加权平均预测分达到8.0时 $collaborative$ 饱和为1.0。$simWeighted = \sum_{}^{}{adjustedSim} \times r(v,d)$ 为邻居对目标菜品的修正相似度加权累积。核心实现逻辑为：遍历 userDishScore 中所有其他用户，跳过自身及未包含目标菜品的用户后，依次计算余弦相似度、统计重叠度、计算收缩修正相似度，将有效邻居的修正相似度加权累加；若最终相似度总量为零则返回0.0，否则依次计算协同预测分、流行度去偏系数和置信度，合成最终得分。

###  4.4.5 融合权重与多餐约束

（1） 融合权重自适应调整

权重公式的设计目标为：冷启动时（signalCount=0）期望PMFUP归一化后权重约0.67；信号极为丰富时（signalCount≥14、coverageSize≥10）期望三权重趋于均衡（各约0.33）。各系数的数值通过这两种极端场景的期望权重反推得到，在5.3节的敏感性分析中验证了±20%系数扰动对推荐排序影响小于 5%。

设用户行为信号数为 $s$，行为覆盖菜品数为 $c$，口味信号总量为 $d$，定义三个中间度量：

权重公式的设计目标为：冷启动时PMFUP归一化后权重约0.67以主导推荐；信号极为丰富时（signalCount≥14、coverageSize≥10）三权重趋于均衡（约各0.33）。各系数通过这两种极端场景的期望权重反推得到。

$$signalConf = clamp\left( \frac{\ln(1 + s)}{\ln 15} \right),$$

$$behavCover = clamp\left( \frac{c}{10.0} \right),$$

$$driftEvid = clamp\left( \frac{d}{12.0} \right)$$

权重原始值的计算公式设计为：PMFUP 基础最高（0.46），随信号增长递减；PRMIDM 受漂移证据驱动；AMMBC 随信号与覆盖度增长。

融合得分为 $S_{fused} = w_{1}S_{pmfup} + w_{2}S_{prmidm} + w_{3}S_{ammbc}$。默认权重 $\lbrack 0.60,0.25,0.15\rbrack$ 体现了\"行为信号缺失时优先依赖画像匹配\"的原则。以新用户冷启动场景为例：$s = 0$、$c = 0$、$d\  \approx 0$，代入得 $signalConf = 0$、$behavCover = 0$、$driftEvid\  \approx 0$，$w_{1}^{raw} = 0.46 + 0.22 + 0.08 = 0.76$，$w_{2}^{raw} = 0.22$，$w_{3}^{raw} = 0.16$，归一化后约为 $\lbrack 0.67,0.19,0.14\rbrack$，接近默认值，PMFUP 主导。当用户积累8条信号、覆盖7种菜品后：$signalConf\  \approx 0.81$，$behavCover\  = \ 0.7$，$w_{1}^{raw} = 0.46 + 0.22 \times 0.19 + 0.08 \times (1 - d)$、$w_{2}^{raw} = 0.22 + 0.30d + 0.12 \times 0.81$、$w_{3}^{raw} = 0.16 + 0.26 \times 0.81 + 0.18 \times 0.7 \approx 0.50$，三权重趋于均衡。当用户行为信号极为丰富（$s\  \geq 14$, $c\  \geq 10$）时，$signalConf\  \approx 1.0$，$behavCover\  = \ 1.0$，AMMBC 的原始权重可增长至 $0.16 + 0.26 + 0.18 = 0.60$，归一化后三个子模型权重大约各占三分之一。这一自适应的设计使得算法无需人工干预即可在冷启动到充分个性化的全生命周期中平滑运行。

### 4.7.3预选定时任务

MealSelectionTask组件包含两个基于Spring \@Scheduled注解的定时任务。自动分配任务（autoSelectWhenOverdue，Cron表达式0 \*/5 \* \* \* ?，每5分钟执行）：扫描所有状态为启用且预选截止时间已过但出发时间未到的航班，对每个航班中尚未手动预选的旅客执行自动菜品分配。分配逻辑为：读取旅客的餐食偏好，查询该航线该舱位下可用的候选菜品列表，按偏好匹配优先级选择菜品，尝试原子扣减库存（SQL UPDATE dish SET stock = stock - 1 WHERE stock \> 0），若扣减成功则写入meal_selection表（状态=5表示自动分配，编号前缀\"AUTO\"），并在推荐日志中写入AUTO_SELECTED_OVERDUE反馈。截止提醒任务（pushSelectionReminder，Cron表达式0 \*/10 \* \* \* ?，每10分钟执行）：对所有启用航班计算距计划起飞时间的剩余小时数。在起飞前12小时和起飞前3小时两个时间节点，自动向航班写入提醒公告（通过flight_announcement表），公告内容包含航班号和预选截止时间提示。旅客可通过小程序公告页查看。

### 4.7.4 评分生命周期管理

评分任务包含四种状态：PENDING（待评分）、SUBMITTED（已提交）、DEFERRED（已暂缓）和EXPIRED（已过期）。航班结束后，系统根据meal_selection中的手动预选记录批量生成PENDING任务（过期时间7天）。旅客提交评分后转为SUBMITTED；暂缓则转为DEFERRED并在24小时后重新提醒；7天内未提交自动过期。管理员可通过后台重开过期任务。为避免同一用户同一航班重复生成评分任务，flight_service_rating 表设置了唯一约束 uk_rating_user_flight(user_id, flight_id)。航班结束后，系统调用 upsertFlightRatingTask 接口，通过 INSERT \... ## 5.3 推荐算法效果分析

### 5.3.1实验数据与评测设置

为系统性地验证算法各组件的功能正确性并分析不同推荐策略的性能边界，设计了两组对照数据集：

**数据集 A（同质数据）**：86 名用户、10 道菜品、4 个航班、309 条推荐日志。用户按舱位分布为头等舱约 10%、商务舱约 10%、经济舱约 80%。10 道菜品覆盖全部 4 种餐食类型和 6 种口味标签，但仅存在 8 种唯一标签组合------2 对菜品标签完全相同（Jaccard=1.00），平均每两道菜品共享 0.53 个标签。数据集 A 的特征是信号同质化程度高、菜品标签区分度较好，代表了 PMFUP 的\"舒适区\"。

**数据集 B（异构数据）**：188 名用户、30 道菜品、2 个航班、812 条推荐日志。在数据集 A 基础上扩充了 100 名用户和 20 道菜品。新增菜品与原有菜品的标签高度重叠（如\"葱油鸡腿饭（咸香,热食,微辣）\"与已有\"红烧牛腩饭（咸香,热食）\"共享 2 个标签，\"蒜蓉西兰花鸡胸饭（清淡,高蛋白,低脂）\"与已有\"低脂鸡胸藜麦饭（清淡,高蛋白,低脂）\"标签完全相同），使得 PMFUP 的标签区分能力显著下降。新增用户分为四组：稳定偏好组（40 人，行为信号一致）、口味漂移组（30 人，历史偏好咸香/微辣，近期偏好清淡/低脂）、偏好邻居组（40 人，集中偏好清真/咸香，形成密集邻居）、冷启动组（10 人，无偏好无历史）。数据集 B 的特征是信号异质化程度高，代表了自适应权重发挥调度价值的\"挑战区\"。

评测方法：将推荐日志按 create_time 升序排列，取后 20% 含 MANUAL_SELECTED 反馈的记录作为测试集。对每个测试样本，以该时间点之前的日志构建交互上下文（避免数据泄露），分别以纯 PMFUP（w=\[[1,0,0]{.underline}\]）、纯 User-CF（w=\[[0,0,1]{.underline}\]）、固定权重（w=\[[0.60,0.25,0.15]{.underline}\]）和自适应权重四种策略调用推荐接口，以用户实际选择的 dishId 作为 ground truth，计算 Top-1 准确率、Top-3 召回率和 MRR。此外，通过亚采样（45/60/全量用户）和 Bootstrap 重采样（200/500/1000 用户）验证性能差距随数据规模的变化趋势。

由表 5.1 可以得出以下观察。第一，纯内容匹配（PMFUP）的 Top-1 准确率（28.6%）在所有策略中最高，而纯协同过滤（User-CF）的 Top-1 仅 9.5%，这表明在 86 人规模下协同过滤几乎无法从邻居信号中提取有效信息------这正是 User-CF 在小样本下的典型表现，验证了本文在冷启动阶段依赖 PMFUP 的设计依据。第二，自适应融合与固定权重融合在 Top-3 召回率上持平（均为 52.4%），但在 Top-1 准确率（19.0% vs 23.8%）和 MRR（0.353 vs 0.377）上均未超越固定权重。这一结果并非对自适应机制的否定，而是表明在当前 86 人、10 道菜品的数据条件下，自适应权重解析器输出的权重向量与默认值 \[0.60, 0.25, 0.15\] 尚未产生显著分离------行为信号的同质化使得\"自适应\"退化为\"固定\"。

为验证上述假设，进一步进行了亚采样分析和 Bootstrap 重采样实验。亚采样分析（表 5.2）表明，在 45 人和 60 人规模下，自适应融合与固定权重的所有指标完全一致（差距为 0.0%），仅当规模扩展至 86 人时 Top-1 出现 4.8 个百分点的分化。Bootstrap 重采样实验进一步将用户规模模拟至 200、500 和 1,000 人，自适应与固定权重的所有指标依然保持完全一致（差距为 0.0%）。这一系列实验共同揭示了一个核心发现：\*\*在当前模拟数据集的同质信号分布下，无论用户规模如何扩大，自适应权重解析器本质上等价于固定权重\*\*。两者的分化需要行为信号的异质性------即用户之间存在足够差异化的交互模式、部分用户具有显著的近期偏好漂移------而当前的模拟数据集尚未构造出此类异质场景。自适应机制的真正价值，预期在候选池扩大使标签区分度下降、用户行为信号充分差异化后，才能通过调度协同过滤和漂移检测来突破内容匹配的性能上限。

第三，自适应融合在异构数据上一致优于固定权重。在亚采样 45/60/90 三档中，自适应融合的 Top-3 召回率均领先固定权重（+6.3%/+11.1%/+6.5%），Bootstrap 重采样中领先幅度为 +17.0% 至 +4.2%。固定权重因 60% 的比重分配给 PMFUP，而在 30 道标签高度重叠的菜品上 PMFUP 几乎失效（Top-1 和 Top-3 均为 0.0%）。自适应融合通过为行为信号丰富的用户提升 PRMIDM/AMMBC 权重，恢复了可测量的推荐能力。

第四，在 Top-5 和 Top-10 召回率上，自适应与固定权重差距缩小。亚采样中两者的 Top-5 和 Top-10 持平（均在 12.5%-22.2% 区间），说明首推的前几位差异已被 30 道菜品的大候选池所稀释。这一现象进一步验证了候选池规模对推荐指标的结构性影响------K 值越大，不同策略之间的区分度越低。

需指出的是，流行度推荐在数据集 B 的所有规模下均取得了最强的 Top-3 召回率（18.8%-27.8%），这表明当前模拟数据中存在曝光-选择的关联偏差------被广泛曝光的菜品恰好也是用户更倾向选择的菜品。这一现象在真实场景中通常因用户偏好的多样性和配餐策略的混杂因素而显著削弱。本文保留这一结果作为基准线，同时将其视为模拟数据局限性的实证证据------未来在真实数据上的验证需要对此关联进行解耦。

#### 5.3.1.3 候选池特征与 PMFUP 优势边界

纯 PMFUP 在所有策略中取得最高 Top-1 准确率（28.6%）的结果，可通过候选菜品池的标签分布特征得到精确解释。对 10 道候选菜品的口味标签相似度分析表明：10 道菜品仅产生 8 种唯一标签组合，45 对菜品组合的平均标签重叠数为 0.53。其中存在 2 对\"标签孪生菜品\"------\"豆浆油条组合\"与\"经典红烧牛腩饭\"的标签组合完全相同（咸香, 热食），\"能量火腿三明治\"与\"香烤鳕鱼时蔬饭\"的标签组合也完全相同（清淡, 高蛋白）------Jaccard 相似度均为 1.00。PMFUP 无法区分这 2 对菜品，但对于其余 6 道具备唯一标签组合的菜品，PMFUP 可以通过餐型（4 类）与口味标签（6 种）的交集匹配实现精确区分。

这一分析揭示了一个具有方法论意义的发现：基于内容标签匹配的推荐策略存在一个由标签空间容量决定的结构性性能上限。当候选菜品数为 10、唯一标签组合数为 8 时，PMFUP 的区分覆盖率高达 80%；但当候选菜品扩展至 50 道时，6 种口味标签的组合空间（理论最大 2⁶ = 64 种，去除不合理组合后实际可用约 30-40 种）将趋于饱和，标签重叠将密集出现，PMFUP 的区分能力将系统性下降。在此条件下，基于用户行为相似性的 PRMIDM（捕捉偏好漂移）和 AMMBC（利用邻域信号）的介入将不再是有害的噪声源，而是突破标签区分瓶颈的必要手段。当前实验受限于 10 道菜品的候选池规模，所有策略的评估均在 PMFUP 的\"舒适区\"内进行------这是自适应机制尚未展现增益的根本结构性原因，也是未来在更大规模候选池上验证算法有效性的明确方向。

# 第 6 章 结论与展望

## 6.1 研究工作总结

本文针对航空餐食推荐场景中供给约束强、偏好动态变化和交互信号稀疏的特征，设计并实现了一个面向航空旅客的智能美食推荐系统。主要工作与发现如下。

（1）领域建模与系统工程。完成了航空餐食推荐场景的领域建模，涵盖航线-菜品绑定、舱位等级递进、多餐时序约束和库存限额等特殊约束，设计了 12 张数据表；基于 Spring Boot 3 + Vue3 + UniApp 技术栈构建了管理端、小程序端和后端三端系统，实现了偏好采集、智能推荐、预选确认、航后评分的完整闭环，共含 70 个 REST API 端点和两个定时任务。

（2）混合推荐方案设计。设计了融合内容匹配（PMFUP）、兴趣漂移检测（PRMIDM）与修正协同过滤（AMMBC）的混合推荐方案。PMFUP 以用户显式偏好进行餐型与口味标签的内容匹配，冷启动可用；PRMIDM 通过对比 30 天近期与 180 天历史行为信号检测口味漂移；AMMBC 在基于用户的协同过滤中引入收缩因子（λ=3.0）、流行度去偏与置信度加权，缓解稀疏数据下的假性高相似与热门偏差问题。三子模型由自适应权重解析器动态融合，并引入前序餐食惩罚机制优化多餐推荐体验。

（3）受控实验与策略适用边界分析。在同质数据集（86 人、10 菜品）和异构数据集（188 人、30 菜品）上进行了系统的对比实验，核心发现包括：（a）自适应权重在同质信号下与固定权重行为等价，两者的分化需要行为信号异质性的引入------数据集 A 中亚采样和 Bootstrap 的所有指标差距均为 0.0%；（b）在异构信号下，自适应融合在所有测试规模上一致优于固定权重------数据集 B Bootstrap 200 人时 Top-3 领先 17.0 个百分点，1000 人时领先 4.2 个百分点；（c）纯 PMFUP 在候选池标签区分度较高时具有结构性优势（数据集 A Top-1 28.6%），但此优势在候选池扩大、标签重叠加剧后完全消失------数据集 B 中固定权重的 Top-3 降至 0%，仅自适应融合仍能维持有效推荐；（d）候选池标签相似度分析揭示了基于标签的内容匹配存在一个由标签空间容量决定的结构性性能上限，当候选菜品扩展至真实场景规模时，行为驱动的协同过滤与漂移检测将成为突破此上限的必要手段。

## 6.2 研究不足与展望

本文工作存在以下不足：（1）实验数据已表明，在 86 人、10 道菜品的小规模模拟数据集上，自适应融合与固定权重的性能在 45 人和 60 人规模下完全一致，仅在 86 人时开始出现分化（Top-1 差距 −4.8%，Top-3 持平），协同过滤的性能受到严重制约（Top-1 仅 9.5%）。亚采样分析初步揭示了自适应机制的性能分化随数据规模增长而加剧的趋势，但其优势拐点尚未达到。候选池特征分析进一步表明，10 道菜品中已存在 2 对标签完全相同的\"孪生菜品\"，当候选池扩大至真实场景的数十道菜品时，标签组合的重叠将更为密集，PMFUP 的区分能力将显著下降，此时 PRMIDM 和 AMMBC 的增量价值才会真正显现。

（2）基于用户的协同过滤在用户规模增大时计算复杂度较高；（3）超参数基于经验设定，缺乏系统化调优；（4）未引入航班时段、季节等上下文特征，偏好建模粒度有待细化；（5）未从精确率、F1-Score 和 AUC 等维度对推荐质量进行系统评估，也未对不同 K 值（如 K=1,3,5,10）下的 Top-K 指标变化规律进行完整分析。此外，2.6 节介绍的 AUC 等排序质量指标受限于模拟数据集无法构造有意义的正负样本分布而未纳入实验。未来工作可从以下方向展开：接入真实航空旅客数据进行算法验证与调优；引入近似最近邻搜索或基于物品的协同过滤优化计算效率；通过网格搜索或贝叶斯优化对超参数进行自动寻优；融合航班时段、航程时长等上下文特征提升偏好建模精度；建立包含离线指标计算和在线A/B实验的完整评价体系。在现有模拟数据上进行亚采样分析（如随机选取 45 人、60 人、90 人三档规模），对比自适应融合与固定权重在不同数据规模下的性能差异，以初步验证\"自适应机制随数据增长而改善\"的趋势假设。本文通过 Bootstrap 重采样将用户规模模拟至 1,000 人，但该方法增加了样本量而未根本改变数据的模拟性质。数据集 B 虽构造了四组异构用户群体以模拟真实场景的信号异质性，但与航空餐食领域的真实旅客行为数据仍存在本质差距。未来需在真实数据上验证本文的受控实验发现。
