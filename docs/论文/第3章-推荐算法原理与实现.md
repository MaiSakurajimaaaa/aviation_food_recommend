# 第3章 推荐算法原理与实现

## 3.1 算法总体设计

航空餐食推荐场景面临三重约束：航班餐食受航线绑定与舱位等级递进规则限制（头等舱可见全部舱位餐食、商务舱可见商务与经济舱、经济舱仅见经济舱）；用户口味存在短期漂移，静态画像难以捕捉；行为信号稀疏且类型多样（曝光、点击、预选、评分等）。针对上述特征，本文提出了一种三子模型融合的自适应混合推荐算法，算法标识为 `fused-pmfup-prmidm-ammbc-v3`。

算法由三个子模型构成：PMFUP（Preference Matching from User Profile）基于用户显式偏好进行内容匹配；PRMIDM（Preference Recent Migration and Interest Drift Model）通过对比近30天与近180天行为信号检测口味漂移；AMMBC（Adaptive Multi-Mode Behavior Collaborative）采用基于用户的协同过滤，引入收缩因子、流行度去偏与置信度加权。三子模型的融合权重由自适应解析器根据用户实时行为信号强度动态计算——信号稀疏时 PMFUP 占主导（默认权重0.60），随信号积累协同过滤与漂移模型权重逐步提升。此外，针对多餐食航班引入前序餐食惩罚机制，避免相邻餐次重复推荐。

算法以一组静态常量集中管理核心参数，包括允许的餐食类型集合 `{1, 2, 3, 4}`（依次对应儿童餐、标准餐、清真餐、素食）、允许的口味标签集合 `{"清淡", "咸香", "微辣", "甜口", "低脂", "高蛋白"}`、舱位等级常量（`CABIN_FIRST=1`, `CABIN_BUSINESS=2`, `CABIN_ECONOMY=3`）、协同过滤收缩系数 `CF_SHRINK_LAMBDA=3.0`、近期窗口 `RECENT_WINDOW_DAYS=30`、历史窗口 `HISTORY_WINDOW_DAYS=180` 以及曝光信号基础权重 `EXPOSURE_SCORE_WEIGHT=0.15`。舱位等级递进规则 `resolveCabinCascade` 的实现逻辑为：若用户舱位为头等舱（1），返回可见类型 `[1, 2, 3]`；若为商务舱（2），返回 `[2, 3]`；若为经济舱（3）或空值，返回 `[3]`。该规则确保高舱位旅客可以浏览低舱位的餐食选项，反之则不可见。候选菜品通过 SQL 查询从 `flight_route_dish` 航线绑定表中筛选，限定条件为航线匹配、舱位等级可见（`cabin_type IN (...)` 递进规则）、库存大于零，单次最多返回20条。算法架构如图3.1所示。

## 3.2 行为数据预处理

### 3.2.1 数据源与行为信号

算法数据源为 `recommendation_log` 表，每次推荐请求时加载近180天内最多5000条日志。每条日志包含：`user_id`（用户ID）、`recommended_dishes`（推荐菜品ID列表，JSON数组）、`user_feedback`（结构化行为反馈字符串）、`user_rating`（航后评分，1-5）和 `create_time`（创建时间）。

`user_feedback` 采用前缀格式记录用户行为：`CLICK:dishId=X:mealOrder=Y` 表示点击浏览，`MANUAL_SELECTED:dishId=X:mealOrder=Y` 表示手动预选，`AUTO_SELECTED_OVERDUE:dishId=X` 表示超时自动分配，`MANUAL_SELECTED_UPDATE:dishId=X:mealOrder=Y` 表示改选。其中 `dishId` 和 `mealOrder` 通过正则匹配从字符串中提取，`dishId` 优先以 `dishId=` 模式匹配，`mealOrder` 以 `mealOrder=` 模式匹配。`recommended_dishes` 字段存储为 JSON 数组（如 `[1, 3, 5]`），算法通过解析 JSON 并结合菜品名称到 ID 的映射表来还原完整的推荐菜品集合。

各类行为按重要性分配差异化的得分权重，并引入指数时间衰减函数加权：

$$
decay(days) = e^{-0.018 \times days}
$$

该函数半衰期约38.5天。表3.1给出了各行为类型的得分权重配置。

### 3.2.2 交互上下文构建

算法通过内部类 `InteractionContext` 在每次请求时批量构建共享数据源，包含六个核心字段：

```java
private static class InteractionContext {
    final Map<Integer, Map<Integer, Double>> userDishScore;      // 用户-菜品累积得分
    final Map<Integer, Integer> dishExposureCount;               // 菜品曝光计数
    final Map<Integer, Integer> userBehaviorSignal;              // 用户行为信号数
    final Map<Integer, Map<String, Double>> userRecentFlavorScore; // 近30天口味得分
    final Map<Integer, Map<String, Double>> userLongFlavorScore;   // 近180天口味得分
    final Map<Integer, Set<String>> dishFlavorTags;              // 菜品口味标签
}
```

其中 `userRecentFlavorScore` 与 `userLongFlavorScore` 以30天为界划分，为 PRMIDM 子模型的漂移检测提供对比基础。`userBehaviorSignal` 统计的是"有行为信号的日志行数"——一条日志无论包含几种行为均计为1。口味标签共有六种：清淡、咸香、微辣、甜口、低脂、高蛋白。

构建过程的核心逻辑以真实源码展示如下：

```java
for (Map<String, Object> row : logs) {
    double decay = timeDecay(parseDateTime(row.get("createTime")));
    // 1) 曝光信号：推荐列表中的每个菜品 +0.15
    for (Integer did : extractDishIds(row.get("recommendedDishes"))) {
        addScore(ctx.userDishScore, uid, did, 0.15 * decay);
        increaseDishExposure(ctx, did);
    }
    // 2) 行为信号：根据feedback前缀分配权重
    Integer selectedDishId = extractDishIdFromText(feedback);
    if (feedback.startsWith("MANUAL_SELECTED")) {
        addScore(ctx.userDishScore, uid, selectedDishId, 5.0 * decay);
        addFlavorDriftSignal(ctx, uid, selectedDishId, createTime, decay, 2.4);
    } else if (feedback.startsWith("AUTO_SELECTED_OVERDUE")) {
        addScore(ctx.userDishScore, uid, selectedDishId, 3.0 * decay);
        addFlavorDriftSignal(ctx, uid, selectedDishId, createTime, decay, 1.8);
    } else if (feedback.startsWith("CLICK")) {
        addScore(ctx.userDishScore, uid, selectedDishId, 2.0 * decay);
        addFlavorDriftSignal(ctx, uid, selectedDishId, createTime, decay, 1.4);
    }
    // 3) 评分信号：≥4分权重2.0，<4分权重1.0
    if (rating != null && rating > 0 && selectedDishId != null) {
        addScore(ctx.userDishScore, uid, selectedDishId, (rating >= 4 ? 2.0 : 1.0) * decay);
        addFlavorDriftSignal(ctx, uid, selectedDishId, createTime, decay, rating >= 4 ? 1.4 : 0.8);
    }
}
```

其中 `addFlavorDriftSignal` 根据 `createTime` 判定归属窗口：距当前30天内的归入 `userRecentFlavorScore`，30至180天内的（或 `createTime` 为空时）归入 `userLongFlavorScore`。其实现逻辑为：根据日志创建时间与当前时间的天数差选择目标 Map，然后遍历菜品的每个口味标签，以 `weight * decay` 为增量累加到目标用户在目标口味上的得分中。`addScore` 采用累加模式（`map.merge(key, delta, Double::sum)`），而非覆盖，使得多次同类行为能够叠加体现持续偏好。辅助函数 `overlapCount` 通过遍历第一个 Map 的键集并检查第二个 Map 是否包含该键来统计共同键数量，在 AMMBC 子模型的收缩修正中用于计算两个用户行为向量的重叠度。

## 3.3 子模型一：PMFUP——基于用户画像的偏好匹配

PMFUP 子模型利用用户显式设定的偏好（餐食类型、口味标签）与菜品属性进行内容匹配，在行为信号稀疏时提供主要的推荐依据。菜品包含两种内容特征：`mealType`（1=儿童餐、2=标准餐、3=清真餐、4=素食）和 `flavorTags`（清淡、咸香、微辣、甜口、低脂、高蛋白中的若干标签）。

评分计算分三步：基础分统一为0.2；若用户偏好集中的餐食类型与菜品匹配，追加奖励——单偏好时追加0.45（明确偏好），多偏好时追加0.38（宽泛偏好）；若口味标签匹配，设匹配数为 $n$，口味奖励为 $\min(0.35, 0.15 + n \times 0.10)$。最终得分钳位至 $[0,1]$：

$$
S_{pmfup} = clamp(0.2 + typeBonus + flavorBonus)
$$

真实源代码如下：

```java
private double calculatePmfupScore(RecommendationDishVO item,
        Set<Integer> prefMealTypes, Set<String> prefFlavors, ...) {
    if (item == null || item.getDishId() == null) return 0.0;
    double prefScore = 0.2;
    if (!prefMealTypes.isEmpty() && item.getMealType() != null
            && prefMealTypes.contains(item.getMealType())) {
        prefScore += prefMealTypes.size() == 1 ? 0.45 : 0.38;
    }
    if (!prefFlavors.isEmpty() && item.getFlavorTags() != null) {
        int matched = 0;
        for (String flavor : prefFlavors) {
            if (item.getFlavorTags().contains(flavor)) matched++;
        }
        if (matched > 0) {
            prefScore += Math.min(0.35, 0.15 + matched * 0.10);
        }
    }
    return clamp01(prefScore);
}
```

PMFUP 得分不依赖航线、时间与行为关联数据，保证了冷启动场景下的可用性。其本质是用户显式声明的偏好与菜品属性标签之间的集合匹配，匹配粒度越细（口味标签重合越多），得分越高，但受上限0.35约束以避免口味匹配过度主导总分。餐食类型匹配的奖励差异化（0.45 vs 0.38）则反映了偏好明确程度的区分：单一偏好意味着用户决策聚焦，匹配应获更高权重。

以具体数值为例：设用户仅偏好标准餐（mealType=2）且偏好口味为 `["清淡", "低脂", "高蛋白"]`（共3种）。对一道标准餐、口味标签为 `["清淡", "低脂", "高蛋白"]` 的"低脂鸡胸藜麦饭"，计算过程为：基础分0.2，餐食类型匹配（单偏好）追加0.45，口味匹配数 $n=3$，口味奖励 $\min(0.35, 0.15+3\times0.10)=0.35$（触及上限），最终得分 $0.2+0.45+0.35=1.0$。对一道标准餐、口味仅为 `["咸香","热食"]` 的"经典红烧牛腩饭"：基础分0.2，类型匹配+0.45，口味匹配数 $n=0$，口味奖励0，最终得分0.65。两者差距体现了口味标签匹配对推荐的区分能力。

## 3.4 子模型二：PRMIDM——偏好漂移与兴趣迁移模型

PRMIDM 子模型通过对比近期（30天）与历史（180天）的口味偏好信号，检测用户口味的短期漂移方向。对于候选菜品 $d$，计算步骤如下。

（1）从交互上下文的 `userRecentFlavorScore` 和 `userLongFlavorScore` 中，汇总用户对菜品所含各口味的近期得分 $R_f$ 与历史得分 $H_f$：

$$
recentScore = \sum_f R_f, \quad historyScore = \sum_f H_f
$$

（2）采用 sigmoid 型归一化函数将两类得分映射至 $[0,1]$。归一化函数的选择需满足三个条件：在 $x=0$ 附近近似线性以保持小信号的区分度，在 $x$ 较大时趋于饱和以避免少数极端值过度支配，值域为 $[0,1)$。本文采用的归一化公式为：

$$
norm(x) = \frac{x}{x + 2.0}
$$

其实现代码为 `private double normalizeSignal(double score) { return clamp01(score / (score + 2.0)); }`，其中常数2.0控制饱和速度：$x=2$ 时归一化值为0.5，$x=8$ 时约为0.8。将两类得分分别归一化：

$$
recentPref = norm(recentScore), \quad historyPref = norm(historyScore)
$$

（3）漂移增量定义为近期偏好减去历史偏好的正向差值：$driftBoost = clamp(recentPref - historyPref)$，仅当近期偏好显著高于历史时才为正。同时获取用户对该菜品的直接兴趣得分 $interest$，计算短期菜品信号 $dishSignal = clamp(interest / 8.0)$。

（4）加权合成：

$$
S_{prmidm} = clamp(0.60 \times recentPref + 0.30 \times driftBoost + 0.10 \times dishSignal)
$$

真实源代码如下：

```java
private double calculatePrmidmScore(Integer userId, Integer dishId, InteractionContext ctx) {
    if (dishId == null || userId == null) return 0.0;
    Set<String> targetFlavors = ctx.dishFlavorTags.getOrDefault(dishId, Collections.emptySet());
    Map<String, Double> recentFlavor = ctx.userRecentFlavorScore.getOrDefault(userId, Collections.emptyMap());
    Map<String, Double> longFlavor = ctx.userLongFlavorScore.getOrDefault(userId, Collections.emptyMap());
    double recentScore = 0.0, historyScore = 0.0;
    for (String flavor : targetFlavors) {
        recentScore += recentFlavor.getOrDefault(flavor, 0.0);
        historyScore += longFlavor.getOrDefault(flavor, 0.0);
    }
    double interest = ctx.userDishScore.getOrDefault(userId, Collections.emptyMap())
            .getOrDefault(dishId, 0.0);
    double recentPref = normalizeSignal(recentScore);
    double historyPref = normalizeSignal(historyScore);
    double driftBoost = clamp01(recentPref - historyPref);
    double shortTermSignal = clamp01(interest / 8.0);
    return clamp01(0.60 * recentPref + 0.30 * driftBoost + 0.10 * shortTermSignal);
}
```

## 3.5 子模型三：AMMBC——自适应协同过滤

AMMBC 子模型采用基于用户的协同过滤，在标准余弦相似度的基础上引入三种修正机制：收缩因子修正小重叠度下的假性高相似，流行度去偏缓解热门菜品的马太效应，置信度加权在邻居证据不足时降低预测可信度。余弦相似度采用单遍点积方式实现，适用于稀疏的用户-菜品得分向量——遍历第一个向量的键集同时累加点积和第一个范数，再单独累加第二个向量的范数，最后以点积除以两个范数平方根的乘积。算法对每个有效邻居（行为向量中包含目标菜品的其他用户）执行以下步骤：计算余弦相似度；统计两用户行为向量的共同键数量作为重叠度；应用收缩因子修正相似度；将修正相似度乘以邻居对该菜品的得分，累加到加权预测值中。

**收缩因子修正**：设两个用户共同交互的菜品数量为 $overlap$，收缩系数 $\lambda = 3.0$，修正后的相似度为：

$$
adjustedSim = sim \times \frac{overlap}{overlap + 3.0}
$$

该修正的效果可通过数值说明：$overlap=1$ 时修正系数为0.25，相似度被大幅削弱，避免因偶然的单一共同菜品产生假性高相似；$overlap=3$ 时修正系数为0.5，相似度折半保留；$overlap=10$ 时修正系数约为0.77，逐步逼近原始相似度。该机制使得两个用户只有在拥有充分共同交互证据时才被认定为高度相似。

**流行度去偏**：设菜品累计曝光次数为 $exp$，惩罚系数为：

$$
popDebias = \frac{1.0}{1.0 + \frac{\ln(1 + exp)}{6.0}}
$$

**置信度加权**：设邻居相似度总量为 $simTotal$，邻居数为 $neighborCount$，置信度定义为：

$$
confidence = clamp\left(\frac{simTotal}{\max(1.0, neighborCount \times 0.45)}\right)
$$

该公式的含义为：当每个邻居的平均修正相似度达到0.45时置信度为1.0；若邻居的平均相似度偏低（如多为0.2左右的弱关联用户），置信度按比例降低。置信度综合反映了"邻居群体在统计意义上是否足够可靠"。例如，3个邻居每人平均相似度仅0.2，则 $simTotal=0.6$，$confidence=0.6/\max(1.0, 3\times0.45)=0.6/1.35\approx0.44$，协同项的权重系数 $0.70+0.30\times0.44=0.83$，协同得分被适度降低。

AMMBC 最终得分由协同预测分与三种修正因子合成：

$$
S_{ammbc} = clamp(collaborative \times popDebias \times (0.70 + 0.30 \times confidence))
$$

其中 $collaborative = clamp(\frac{simWeighted / simTotal}{8.0})$，系数8.0为归一化缩放因子——当加权平均预测分达到8.0时 $collaborative$ 饱和为1.0。$simWeighted = \sum adjustedSim \times r(v,d)$ 为邻居对目标菜品的修正相似度加权累积。核心实现逻辑为：遍历 `userDishScore` 中所有其他用户，跳过自身及未包含目标菜品的用户后，依次计算余弦相似度、统计重叠度、计算收缩修正相似度，将有效邻居的修正相似度加权累加；若最终相似度总量为零则返回0.0，否则依次计算协同预测分、流行度去偏系数和置信度，合成最终得分。

## 3.6 自适应权重融合与多餐约束

### 3.6.1 融合权重自适应调整

设用户行为信号数为 $s$，行为覆盖菜品数为 $c$，口味信号总量为 $d$，定义三个中间度量：

$$
signalConf = clamp\left(\frac{\ln(1 + s)}{\ln 15}\right), \quad
behavCover = clamp\left(\frac{c}{10.0}\right), \quad
driftEvid = clamp\left(\frac{d}{12.0}\right)
$$

权重原始值的计算公式设计为：PMFUP 基础最高（0.46），随信号增长递减；PRMIDM 受漂移证据驱动；AMMBC 随信号与覆盖度增长。真实源码如下：

```java
private double[] resolveFusionWeights(Integer userId, InteractionContext ctx) {
    if (userId == null || ctx == null) return new double[]{0.60, 0.25, 0.15};
    int signalCount = ctx.userBehaviorSignal.getOrDefault(userId, 0);
    double signalConf = clamp01(Math.log1p(signalCount) / Math.log(15.0));
    int coverageSize = ctx.userDishScore.getOrDefault(userId, Collections.emptyMap()).size();
    double behavCover = clamp01(coverageSize / 10.0);
    double recentSig = sumSignal(ctx.userRecentFlavorScore.getOrDefault(userId, Collections.emptyMap()));
    double historySig = sumSignal(ctx.userLongFlavorScore.getOrDefault(userId, Collections.emptyMap()));
    double driftEvid = clamp01((recentSig + historySig) / 12.0);
    double w1 = 0.46 + 0.22 * (1 - signalConf) + 0.08 * (1 - driftEvid);
    double w2 = 0.22 + 0.30 * driftEvid + 0.12 * signalConf;
    double w3 = 0.16 + 0.26 * signalConf + 0.18 * behavCover;
    double total = w1 + w2 + w3;
    if (total <= 0) return new double[]{0.60, 0.25, 0.15};
    return new double[]{w1 / total, w2 / total, w3 / total};
}
```

融合得分为 $S_{fused} = w_1 S_{pmfup} + w_2 S_{prmidm} + w_3 S_{ammbc}$。默认权重 $[0.60, 0.25, 0.15]$ 体现了"行为信号缺失时优先依赖画像匹配"的原则。以新用户冷启动场景为例：$s=0$、$c=0$、$d \approx 0$，代入得 $signalConf=0$、$behavCover=0$、$driftEvid \approx 0$，$w_1^{raw}=0.46+0.22+0.08=0.76$，$w_2^{raw}=0.22$，$w_3^{raw}=0.16$，归一化后约为 $[0.67, 0.19, 0.14]$，接近默认值，PMFUP 主导。当用户积累8条信号、覆盖7种菜品后：$signalConf \approx 0.81$，$behavCover = 0.7$，$w_1^{raw}=0.46+0.22\times0.19+0.08\times(1-d)$、$w_2^{raw}=0.22+0.30d+0.12\times0.81$、$w_3^{raw}=0.16+0.26\times0.81+0.18\times0.7\approx0.50$，三权重趋于均衡。当用户行为信号极为丰富（$s \geq 14$, $c \geq 10$）时，$signalConf \approx 1.0$，$behavCover = 1.0$，AMMBC 的原始权重可增长至 $0.16+0.26+0.18=0.60$，归一化后三个子模型权重大约各占三分之一。这一自适应的设计使得算法无需人工干预即可在冷启动到充分个性化的全生命周期中平滑运行。

> **[此处插入三线表3.2：不同行为信号强度下的融合权重示例]**
> | 场景 | signalCount | 覆盖菜品数 | PMFUP权重 | PRMIDM权重 | AMMBC权重 |
> |------|:---------:|:--------:|:--------:|:---------:|:--------:|
> | 新用户冷启动 | 0 | 0 | ~0.60 | ~0.25 | ~0.15 |
> | 少量交互 | 3 | 3 | ~0.48 | ~0.28 | ~0.24 |
> | 中等交互 | 8 | 7 | ~0.39 | ~0.31 | ~0.30 |
> | 丰富交互 | 15+ | 10+ | ~0.33 | ~0.33 | ~0.34 |

### 3.6.2 多餐前序惩罚与解释性输出

对于第 $n$ 餐（$n \geq 2$），查询用户在前序餐次 $n-1$ 的选择记录。实现方式为：通过 `recommendationMapper.latestManualSelection(userId, flightId, mealOrder-1)` 获取前序餐次的预选记录，提取 `prevDishId` 及其 `mealType`，然后在逐菜品评分循环中判断——若当前菜品ID等于 `prevDishId` 则乘以0.70，否则若菜品 `mealType` 与 `prevMealType` 相同则乘以0.85。该机制不阻断任何菜品的推荐——惩罚后仍可出现在推荐列表中（仅为降权），在"避免重复"与"保留选择权"之间取得平衡。

## 3.7 算法总体流程

综合以上各节，算法完整流程的伪代码如算法1所示。

**算法1：Fused-PMFUP-PRMIDM-AMMBC 混合推荐算法**

```
输入: userId, flightId, mealOrder
输出: 个性化推荐列表 (含得分与解释标签)

 1: flight ← getFlightInfo(flightId)
 2: cabinTypes ← resolveCabinCascade(user.cabinType)
       // 头等舱→[1,2,3], 商务舱→[2,3], 经济舱→[3]
 3: candidates ← listCandidateDishes(flight.departure, flight.destination,
       cabinTypes, limit=20)  // SQL查询，含航线绑定与库存过滤
 4: if candidates为空: return []
 5:
 6: logs ← listRecentLogs(180天, 5000条)
 7: context ← buildInteractionContext(logs)     // 3.2节, 构建共享数据源
 8: [w1,w2,w3] ← resolveFusionWeights(userId, context)  // 3.6.1节, 自适应权重
 9: prevDish ← findPreviousSelection(userId, flightId, mealOrder-1)
10:
11: for each dish in candidates:
12:     S1 ← calculatePmfupScore(dish, userPrefs)    // 3.3节
13:     S2 ← calculatePrmidmScore(userId, dish.id, context)  // 3.4节
14:     S3 ← calculateAmmbcScore(userId, dish.id, context)   // 3.5节
15:     fused ← w1*S1 + w2*S2 + w3*S3
16:     fused ← applySequentialPenalty(fused, dish, prevDish)  // 3.6.2节
17:     dish.score ← clamp01(fused)
18:     dish.reason ← generateExplain(S1, S2, S3, signalCount) // 3.6.2节
19:
20: sort candidates by score descending
21: insertRecommendationLog(userId, flightId, candidates)
22: return candidates
```

时间复杂度分析：候选查询阶段通过单次 SQL 完成，复杂度 $O(|D|)$；上下文构建阶段遍历最多5000条日志及每条日志的推荐列表，复杂度 $O(|L| \times |R|)$；评分阶段中，PMFUP 为 $O(|C|)$，PRMIDM 为 $O(|C| \times |F|)$（$|F| \leq 6$），AMMBC 需遍历所有用户计算余弦相似度，为主要计算开销 $O(|U| \times |D_u|)$。在90用户、10菜品的测试数据集上，单次推荐延时为毫秒级，满足实时要求。算法中复用的四个辅助函数均为无外部库依赖的简单实现：`timeDecay` 计算指数衰减 $e^{-0.018 \times days}$（时间为空时返回0.8）；`clamp01` 将值钳位至 $[0,1]$；`sumSignal` 对口味得分 Map 中的所有正值求和（用于权重解析时计算漂移证据总量）；`normalizeSignal` 执行 sigmoid 型归一化 $x/(x+2.0)$（用于 PRMIDM 的偏好得分标准化）。
